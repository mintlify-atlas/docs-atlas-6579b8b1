This guide walks you through starting a SPIRE Server, joining an Agent, registering a workload entry, and fetching an X.509-SVID — the end-to-end flow on a single Linux host.

<Note>
  These steps use the default `join_token` node attestor and a SQLite datastore, which are suited for local testing. See [Installation](/deployment/installation) for production-ready configuration.
</Note>

<Steps>

  <Step title="Download SPIRE">
    Download the latest pre-built release from the [SPIRE releases page](https://github.com/spiffe/spire/releases). Each release contains both the `spire-server` and `spire-agent` binaries.

    ```bash
    # Replace X.Y.Z with the latest release version
    curl -Lo spire-X.Y.Z-linux-amd64-musl.tar.gz \
      https://github.com/spiffe/spire/releases/download/vX.Y.Z/spire-X.Y.Z-linux-amd64-musl.tar.gz

    tar xzf spire-X.Y.Z-linux-amd64-musl.tar.gz
    cd spire-X.Y.Z
    ```

    The archive unpacks into a directory containing `bin/spire-server`, `bin/spire-agent`, and example configuration files under `conf/`.
  </Step>

  <Step title="Start the SPIRE Server">
    Create a minimal server configuration file, then start the server.

    ```hcl
    # conf/server/server.conf
    server {
        bind_address = "127.0.0.1"
        bind_port = "8081"
        socket_path = "/tmp/spire-server/private/api.sock"
        trust_domain = "example.org"
        data_dir = "./.data"
        log_level = "DEBUG"
    }

    plugins {
        DataStore "sql" {
            plugin_data {
                database_type = "sqlite3"
                connection_string = "./.data/datastore.sqlite3"
            }
        }

        NodeAttestor "join_token" {
            plugin_data {
            }
        }

        KeyManager "memory" {
            plugin_data = {}
        }
    }
    ```

    ```bash
    ./bin/spire-server run -config conf/server/server.conf &
    ```

    The server starts on `127.0.0.1:8081` and creates its SQLite datastore under `.data/`. You should see `"SPIRE Server is ready"` in the log output.

    <Tip>
      For a complete list of server configuration options, see [Server configuration](/configuration/server).
    </Tip>
  </Step>

  <Step title="Generate a join token">
    Generate a one-time join token that the agent will use to attest itself to the server. The `-spiffeID` flag associates a SPIFFE ID with the node after attestation.

    ```bash
    ./bin/spire-server token generate \
      -spiffeID spiffe://example.org/host
    ```

    The command prints a token similar to:

    ```text
    Token: 7f6fc6b8-4b88-4e5c-9e45-1c2a3d4e5f6a
    ```

    Copy this token — you will need it in the next step. By default the token expires after 600 seconds.
  </Step>

  <Step title="Start the SPIRE Agent">
    Create an agent configuration file that points at the server and includes the trust bundle, then start the agent with the join token from the previous step.

    ```hcl
    # conf/agent/agent.conf
    agent {
        data_dir = "./.data"
        log_level = "DEBUG"
        server_address = "127.0.0.1"
        server_port = "8081"
        socket_path = "/tmp/spire-agent/public/api.sock"
        trust_bundle_path = "./conf/agent/dummy_root_ca.crt"
        trust_domain = "example.org"
    }

    plugins {
        NodeAttestor "join_token" {
            plugin_data {
            }
        }
        KeyManager "disk" {
            plugin_data {
                directory = "./.data"
            }
        }
        WorkloadAttestor "unix" {
            plugin_data {
            }
        }
    }
    ```

    ```bash
    ./bin/spire-agent run \
      -config conf/agent/agent.conf \
      -joinToken <generated-join-token> &
    ```

    The agent connects to the server, attests using the join token, receives the trust bundle, and begins exposing the Workload API socket at `/tmp/spire-agent/public/api.sock`.

    <Tip>
      For a complete list of agent configuration options, see [Agent configuration](/configuration/agent).
    </Tip>
  </Step>

  <Step title="Create a workload registration entry">
    Register a workload by mapping a set of selectors to a SPIFFE ID. This entry tells the server which processes are allowed to receive the identity `spiffe://example.org/workload`.

    The example below uses a Unix UID selector — the SVID will be issued to any process running as UID `1001`.

    ```bash
    ./bin/spire-server entry create \
      -parentID spiffe://example.org/host \
      -spiffeID spiffe://example.org/workload \
      -selector unix:uid:1001
    ```

    A successful registration prints the entry ID and the details of the new entry.

    <Note>
      The `-parentID` must match the SPIFFE ID assigned to the node when the join token was generated. The agent will only issue SVIDs for entries whose parent ID matches the node's own SPIFFE ID.
    </Note>
  </Step>

  <Step title="Fetch an X.509-SVID">
    Simulate workload API access by running `spire-agent api fetch x509` as the user with UID `1001`. The agent checks the caller's identity against the registered entry and returns an X.509-SVID.

    ```bash
    # Run as the workload user (uid 1001)
    su -c "./bin/spire-agent api fetch x509" workload
    ```

    The output shows the SVID certificate chain and the trust bundle:

    ```text
    Received 1 svid after 5.0338ms

    SPIFFE ID:         spiffe://example.org/workload
    SVID Valid After:  2024-01-01 00:00:00 +0000 UTC
    SVID Valid Until:  2024-01-01 01:00:00 +0000 UTC
    CA #1 Valid After: 2024-01-01 00:00:00 +0000 UTC
    CA #1 Valid Until: 2025-01-01 00:00:00 +0000 UTC
    ```

    To write the SVID and private key to disk for inspection:

    ```bash
    su -c "./bin/spire-agent api fetch x509 -write /tmp" workload
    openssl x509 -in /tmp/svid.0.pem -text -noout
    ```

    The certificate's Subject Alternative Name contains the SPIFFE URI `spiffe://example.org/workload`, confirming that the identity was correctly issued.
  </Step>

</Steps>
