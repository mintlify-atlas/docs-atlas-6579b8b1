SPIRE Server and Agent can emit metrics to one or more collectors. Metrics are defined in a dedicated `telemetry { ... }` block in the configuration file and are separate from the regular log output.

## Supported backends

<CardGroup cols={3}>
  <Card title="Prometheus" icon="chart-line">
    Exposes a scrape endpoint. Supports optional TLS and mTLS.
  </Card>
  <Card title="Statsd" icon="signal">
    Pushes metrics to one or more Statsd addresses. Multiple declarations are supported.
  </Card>
  <Card title="DogStatsd" icon="dog">
    Pushes metrics to one or more DogStatsd addresses. Multiple declarations are supported.
  </Card>
  <Card title="M3" icon="database">
    Pushes metrics to one or more M3 collectors. Multiple declarations are supported.
  </Card>
  <Card title="In-Memory" icon="memory">
    Stores metrics in memory. Enabled by default; primarily useful for testing.
  </Card>
</CardGroup>

You can configure any combination of these backends simultaneously. Statsd, DogStatsd, and M3 all accept multiple address entries so you can fan out to more than one collector of the same type.

## Configuration

Add a `telemetry { ... }` block to your SPIRE Server or Agent configuration file. The block is identical in structure for both components.

### Full example

```hcl
telemetry {
  Prometheus {
    port = 9988
    # Optional TLS for the Prometheus scrape endpoint
    tls {
      cert_file      = "/path/to/cert.pem"
      key_file       = "/path/to/key.pem"
      client_ca_file = "/path/to/ca.pem"  # optional — enables mTLS
    }
  }

  DogStatsd = [
    { address = "localhost:8125" },
  ]

  Statsd = [
    { address = "localhost:1337" },
    { address = "collector.example.org:8125" },
  ]

  M3 = [
    { address = "localhost:9000" env = "prod" },
  ]

  InMem {}

  AllowedLabels   = []
  BlockedLabels   = []
  AllowedPrefixes = []
  BlockedPrefixes = []
}
```

### Global options

| Option | Type | Description | Default |
|---|---|---|---|
| `MetricPrefix` | string | Prefix prepended to all emitted metric names. | `spire_server` / `spire_agent` |
| `EnableTrustDomainLabel` | bool | Attach the trust domain as a label on every metric. | `false` |
| `EnableHostnameLabel` | bool | Attach the hostname as a label on every metric. | `true` |
| `AllowedPrefixes` | list | Only emit metrics whose name starts with one of these prefixes (`.` separator). | (all allowed) |
| `BlockedPrefixes` | list | Drop metrics whose name starts with one of these prefixes (`.` separator). | (none blocked) |
| `AllowedLabels` | list | Only emit these label keys. | (all allowed) |
| `BlockedLabels` | list | Drop these label keys from all metrics. | (none blocked) |

### Prometheus options

| Option | Type | Description |
|---|---|---|
| `host` | string | Listen address for the Prometheus scrape endpoint. |
| `port` | int | Listen port for the Prometheus scrape endpoint. |

### DogStatsd options

| Option | Type | Description |
|---|---|---|
| `address` | string | DogStatsd agent address (host:port). |

### Statsd options

| Option | Type | Description |
|---|---|---|
| `address` | string | Statsd server address (host:port). |

### M3 options

| Option | Type | Description |
|---|---|---|
| `address` | string | M3 collector address (host:port). |
| `env` | string | M3 environment tag, e.g. `production` or `staging`. |

## Metric format

Metrics are identified by a sequence of keys and optional labels. The exact wire format depends on the backend. For example, a successful SPIRE Server `AttestAgent` RPC in Statsd format looks like:

```text
spire_server.rpc.agent.v1.agent.attest_agent:1|c|#status:OK
spire_server.rpc.agent.v1.agent.attest_agent.elapsed_time:1.045773|ms|#status:OK
```

The hostname and service name are appended by the Statsd backend automatically.

### Call counters

Call counters are a composite metric type emitted around any method or RPC invocation. For each call counter, SPIRE emits:

- A **counter** tracking the number of invocations (key as-is).
- A **sample** measuring the elapsed time (key + `.elapsed_time`).

Both carry a `status` label holding the [gRPC status code](https://pkg.go.dev/google.golang.org/grpc/codes#Code) of the call.

## Grafana dashboards

A pre-built Grafana dashboard JSON is included in the SPIRE repository at [`doc/telemetry/spire_grafana_dashboard.json`](https://github.com/spiffe/spire/blob/main/doc/telemetry/spire_grafana_dashboard.json). It covers server and agent request rates, latency, CA rotation status, cache sizes, and runtime information.

<Steps>
  <Step title="Configure Prometheus in SPIRE">
    Add a `Prometheus` block to your `telemetry` section and point your Prometheus server at the scrape endpoint.
  </Step>
  <Step title="Add Prometheus as a Grafana data source">
    In Grafana, navigate to **Configuration → Data sources** and add your Prometheus instance. Note the data source UID — the dashboard expects `uid_prometheus_grafana_datasource`.
  </Step>
  <Step title="Import the dashboard">
    In Grafana, go to **+ → Import dashboard** and upload `spire_grafana_dashboard.json` from the repository.
  </Step>
</Steps>

<Tip>
  Use the `AllowedPrefixes` and `BlockedPrefixes` options to reduce cardinality if your metrics backend charges by series count. For example, blocking `datastore` prefixes removes the many fine-grained datastore call counters if you only need higher-level visibility.
</Tip>
